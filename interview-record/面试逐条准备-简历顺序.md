## 条目 1 — Agent Harness 分层设计

**简历原文**：参考 pi / Claude Code 架构，将建模工具抽象为 Session、Runtime State、Tool Registry、Event Stream 四层，沉淀出可编排、可复用、可观测的 Tool API。

### 真实实现校准

四层能对应到真实类，抽象站得住，但要能当场映射（别只背名词）：

| 简历层 | 真实类 | 职责 |
|---|---|---|
| Session | `session/PtolemySession` + `SessionManager` | 一个 sid 一个独立 `Workspace + MoMLParser + CompositeActor + Manager` + 私有注入的 Recorder |
| Runtime State | `session/ModelContext` | 把当前模型压成**紧凑结构化 JSON**（nodes/edges/directors/summary + classFailures），喂给 LLM，"prefer this over raw MoML" |
| Tool Registry | `tools/ToolRegistry` | 注册 14 个 `AgentTool`，`dispatch(name, session, args)` 统一入口，`toolsForOpenAI()` 出 function-calling schema |
| Event Stream | `agent/AgentTrace` + `AgentTraceListener` | 每步一个 `Step{index, kind, name, arguments, text, timestamp}`，kind ∈ thought/tool_call/tool_result/final/error，listener 实时 `onStep` → NDJSON 推前端 |

**可编排/可复用/可观测**的真实证据：
- 可编排：`ToolRegistry.only(...)` / `except(...)` 给不同阶段裁剪工具子集（builder 隐藏 composite 工具，refactor 只留 group 工具）。
- 可复用：同一个 `dispatch` 被 LLM、`PlanExecutor`、HTTP 直调、前端拖拽**四方复用**，走同一 preflight、同一 trace。
- 可观测：`AgentTrace` step 流 + `diagnostics`。

### 必备知识点（理论纵深）

- **什么是 Agent Harness**：把"LLM + 工具 + 状态 + 事件"从业务逻辑里剥离出来的运行骨架。pi / Claude Code 的共性抽象 = Session（会话与隔离）/ Tool（能力）/ Loop（ReAct 循环）/ Trace（可观测）。
- **Tool 抽象的关键**：name + description + JSON Schema（parameters）三件套 → 这正是 OpenAI function calling 的协议格式（`{"type":"function","function":{name, description, parameters}}`）。要能讲清 LLM 是"看 description + schema 决定调哪个工具、怎么填参数"。
- **分层的收益**：关注点分离、可测试（每层可 mock）、可替换（换 LLM/换宿主系统只动一层）。
- **对标主流框架**：LangChain 的 Tool/AgentExecutor、LangGraph 的 StateGraph+节点、MCP 的 client-server 工具协议。你的 `ToolRegistry` ≈ 这些框架的工具注册中心；`AgentTrace` ≈ LangGraph 的 state/checkpoint 可观测层。

### 高频追问

- **Q：为什么要自己抽象四层而不用 LangChain？**
  A：这是嵌在 Ptolemy II（Java）里的领域系统，要贴 MoML/Manager 执行语义、内核零侵入、同一 Tool API 服务 LLM/HTTP/UI 四方。用 Python 框架反而要架桥。但设计模式借鉴了它们。
- **Q：Runtime State 为什么不直接把 MoML（XML）喂给 LLM？**
  A：MoML 冗长、截断后语义丢失。`ModelContext` 做成紧凑语义 JSON（actor/port/连接/director/计数），token 更省、LLM 更好推理。这本身就是**上下文工程**（结构化替代原始长文本）。
- **Q：Event Stream 怎么做到"可观测"？持久化了吗？** ⚠️
  A：诚实说——trace 是**会话内内存对象 + 序列化回传/NDJSON 流式**，不是落盘持久化（曾有 `CheckpointStore` 写 MoML 快照到磁盘，但随 MEGA 一起删了）。要说"可观测=step 流 + replay，durable 持久化是待补项"。

职责 1（架构设计）、职责 4（可复用组件/Tool API 沉淀）、加分项 4（框架二次开发）。

## 条目 2 — 建模运行时建设

**简历原文**：落地会话隔离、任务状态机、工具调用上下文与结果持久化，支撑建模/评测任务的全链路追踪、复现与调试回放。

> **核心区分（若被问"这条和条目 1 不重复吗"）**：条目 1 是**静态架构**（系统怎么搭、由哪几层组成），条目 2 是**动态运行时**（任务跑起来时怎么隔离、状态怎么流转、变更怎么撤销、出错怎么回放）。一个是名词/结构，一个是动词/行为。代表代码：条目 1 看 `ToolRegistry`/`AgentTrace`（结构），条目 2 看 `PtolemySession` 的 applyChange/run/undo/dryRun/_state（行为）。

### 真实实现校准（读完 `PtolemySession` 784 行后修正——比初版更能打）

| 简历说法 | 真实实现 | 校准 |
|---|---|---|
| 会话隔离 | `PtolemySession` 每 sid 建**独立 `Workspace`**（`new Workspace("AgentSession-"+id)`）+ 独立 MoMLParser/Manager；所有 mutating 方法 `synchronized` | ✅ 且有深度：代码注释点明**独立 workspace 是为规避 Ptolemy 全局静态 filter registry 的跨会话串扰**——体现读懂了宿主框架并发模型 |
| 任务状态机 | `PtolemySession._state`：`IDLE→READY→RUNNING→FINISHED/ERROR`(+UNKNOWN)，由 `ExecutionListener` 回调（`managerStateChanged`/`executionFinished`/`executionError`）驱动，镜像 Ptolemy `Manager` 执行状态机 | ✅ 真实存在（**修正初版过度保守**）。说法：一个"执行生命周期状态机"，复用/映射 Manager 状态，**不夸大成"自研 FSM 引擎"** |
| 工具调用上下文与结果持久化 | ① `applyChange()` 用 `MoMLChangeRequest` + `setUndoable(true)` → `undo()/redo()`（Ptolemy `UndoStackAttribute`）；② `saveToFile()/exportMoml()` 模型落盘为 MoML（先摘探针 Recorder 保证干净）；③ `classFailures` 会话内失败上下文；④ `AgentResult` 结构化结果 | ✅ 比初版强：**模型态可落盘 + 变更可撤销回退**（修正"无 durable 落盘"）。准确表述：模型态可持久化到 MoML、变更可撤销；执行轨迹为会话内内存对象并流式回传 |
| 全链路追踪/复现/回放 | `AgentTrace` 事件流 + **`dryRunTool()` 克隆体预览基元**（在临时 copy 上执行工具+validate，返回 toolResult/validation/新 modelContext，`committed=false`，不碰真模型）+ `undo()/redo()` | ✅ trace/undo 真实在用。⚠️ **`dryRunTool` 是"已实现但未接线"的 latent primitive**（全仓库无调用方设 flag），是**预览基元非事务**（只有预览半、无 commit 半，见条目 3）。"复现"= trace 回放 + undo 回退（+ dry-run 预览能力，但当前休眠），非字节级确定性重放（LLM 采样随机） |

### 必备知识点（理论纵深）

- **会话隔离的意义**：多用户/多任务并发时状态不串扰；每个 session 是独立可变状态容器。你这里的深度点是**识别到宿主框架的全局静态状态陷阱（filter registry），用 per-session Workspace 隔离掉**。对应记忆系统里的"工作记忆边界"。
- **执行状态机 vs 编排流程（两个层次别混）**：
  - 执行状态机 = 单次仿真的生命周期（`_state`: IDLE/RUNNING/FINISHED/ERROR），由 Manager 回调驱动。
  - 编排流程 = 多角色阶段推进（`AgentPipeline` 的 Phase0→0.5→1→2→2.5）+ `AgentLoop` 的 progress/stall 跟踪（ReAct 循环）。
  - 面试要能分清：前者是"任务跑没跑完/出没出错"，后者是"多智能体走到哪一工序"。
- **状态机 vs ReAct 循环（两种编排范式）**：Pipeline 阶段固定、可预测、易观测；ReAct think→act→observe 灵活但可能失控（所以配反振荡/stall/步数上限）。确定性流程用 Pipeline，开放式修补用 ReAct。
- **沙盒预览（dry-run）思想**：在**克隆体**上试跑一个操作、看结果和校验、再决定要不要在真身上提交。这是"安全执行 + 调试预览"的通用模式（类比 `kubectl apply --dry-run`、git 试验分支）。⚠️ 你的 `dryRunTool` 实现了这个模式，但**只是"预览半"、无 commit 半，且当前未接到任何调用方**（详见条目 3 的 dry-run 专条）。讲它=讲设计能力，别讲成"在用的事务功能"。
- **可撤销变更（undo 栈）**：把每次 mutation 做成可逆操作（命令模式 Command Pattern），是编辑器/建模器的标配，也是"调试回放"的基础。
- **可观测性三件套**：Trace（步骤链，已做）/ Metrics（成功率、步数、token，待补）/ Log（`_logMetric` 打了耗时日志）。
- **复现/回放的价值与边界**：调试 Agent 最难的是"复现失败"。trace+dry-run+undo 让你能回看和试验，但 LLM 采样随机，严格复现需固定 seed/temperature。

### 高频追问

- **Q：你的"任务状态机"有哪些状态、怎么流转？**
  A（准确版）："每个会话维护一个执行生命周期状态机 IDLE→READY→RUNNING→FINISHED/ERROR，由 Ptolemy Manager 的 ExecutionListener 回调驱动流转，`status()` 对外暴露，前端/调用方据此判断进展。它复用了 Manager 的状态语义，我在会话层做了缓存和暴露，而不是从零造一个 FSM 引擎。"
- **Q：会话隔离具体隔离了什么？为什么要独立 Workspace？**
  A："每个会话独立的 Workspace + MoMLParser + Manager + toplevel。关键是 Ptolemy 的 MoML filter 是全局静态注册的，多会话共享会串扰，所以必须 per-session Workspace 隔离；所有 mutating 方法 synchronized 保证单会话内线程安全。"
- **Q：怎么做到"复现与调试回放"？**
  A："主要靠两件在用的：① `AgentTrace` 记录每步（thought/tool_call/tool_result/final/error）可回看；② `undo/redo` 单步回退。另外我实现了 `dryRunTool` 预览基元（克隆模型试跑+validate 不提交），是为安全执行/what-if 设计的，但当前还没接到调用方。注意 LLM 有随机性，所以是行为回放，不是字节级确定性重放。"
- **Q：dryRun 是怎么实现"不影响真实模型"的？**
  A："`exportMoml()` 导出当前模型 → 在一个新建的临时 `PtolemySession` 上 `loadMoml` 重建 → 在这个 copy 上 dispatch 工具 + validate → 返回结果后 `disposeModel()` 销毁 copy。真身全程只读。"
- **Q：结果/上下文持久化到什么程度？**
  A（诚实且准确）："模型态可 `saveToFile` 落盘为 MoML、变更进 undo 栈可回退、会话内维护 classFailures 失败上下文；执行 trace 是会话内内存对象 + NDJSON 流式回传，跨进程 durable 的 trace 存储是待补项（曾有 CheckpointStore 落盘快照，随 mega 一起回退了）。"
- **Q：怎么支撑"评测任务"？** ⚠️
  A：诚实但已补了一步——除了 5 个 smoke test + 3 个 demo，我另外搭了一个**小型内部 smoke-eval**（`eval/Run-AgentEval.ps1`）：在一组 canonical 建模任务上跑 **pipeline vs single 对照**，从 `AgentTrace` + 一次客观 `/run` 里解析出「可运行模型率、平均工具调用数、失败调用数、反振荡命中、builder 跳过率、时延」。**明确说清边界**：样本小 + LLM 随机性，是方向性证据不是发表级 benchmark；token 接口拿不到；逐层防线消融需要加 feature flag 才能做。别夸大成"完整评测体系"。

## 条目 3 — Tool Calling 可靠性治理（★ 最强项，重点准备）

**简历原文**：设计工具级参数校验、幂等控制与批量事务回滚，实现 didYouMean 端口建议、实体校验、方向检测等智能纠错，错误上下文精确回传以降低 Agent 无效重试。

### 真实实现校准（全部有代码，可大胆讲）

| 简历点 | 真实类/机制 |
|---|---|
| 参数校验 | `ToolCallValidator.validate()` 在 `dispatch` 里 preflight，所有工具共用 |
| 幂等控制 | `AddEntityTool`/`SetParameterTool` 同名同类/同值再调 → `ok=true, noop=true` |
| 批量事务回滚 ⚠️ | `ConnectManyTool`：全 preflight，任一错→整批拒绝、一条不落；全过→拼一条 MoML `<group>` **原子提交**（撤销栈看到单次操作）。⚠️ 严格说是 **abort-before-commit（提交前中止）+ 原子 group 提交**，不是"写一半再撤"；是**工具级/单进程原子性，非 ACID/分布式事务**。"事务回滚"这个词锚定到 `connect_many`，**别锚定到 dry-run** |
| didYouMean | `PortHints`：大小写不敏感 Levenshtein + 分桶排序，top-3 |
| 实体校验 | `PortHints.suggestEntities/suggestComposites/suggestAttributes` |
| 方向检测 | `_directionMatches` + 边界端口 input/output 语义翻转 + `REVERSED_DIRECTION` 单独识别，返回 `actualDirection` |
| 错误上下文精确回传 | 错误从 `{"ok":false,"message":"port not found"}` 升级为带 `code/didYouMean/availablePorts/actualDirection`，且**第一条建议嵌进 message 文本**（LLM 不解析结构字段也能看到） |
| 降低无效重试 | 上面全部 + `ToolRegistry` 失败归因 `classFailures`（≥2 次同类失败 → 提示换 alternative） |

### 必备知识点（理论纵深）

- **为什么 Tool Calling 可靠性是 Agent 工程的核心**：LLM 幻觉 = 编造不存在的类名/端口/参数。工具层是"现实世界的边界"，必须把幻觉挡在这里，否则错误会级联污染后续步骤。
- **Levenshtein（编辑距离）**：两字符串最少的增/删/改次数。两行滚动数组 DP，O(n·m) 时间 O(min) 空间。你要能手写核心递推：`dp[i][j] = min(删, 增, 改)`，改的代价 = 字符相等?0:1。
- **模糊纠错的分层排序思想**：精确匹配 > 前后缀 > 子串 > 纯编辑距离。避免短名字上的噪声（bucket≥3 且距离>3 才丢）。
- **幂等（Idempotency）**：同一操作执行多次效果等同一次。分布式/重试场景的基石。你的实现是"检测到同名同值就 noop"。要能对比 HTTP PUT（幂等）vs POST（非幂等）。
- **事务原子性（Atomicity）与"你到底有没有事务"**：⚠️ **你没有通用事务功能（没有跨多操作的 begin/commit/rollback 边界，没有事务管理器）**。你有的是几个"事务式"性质，面试要能精确列出并划边界：
  - 单次变更原子性：`applyChange` 的 `MoMLChangeRequest`，Ptolemy 要么 changeExecuted 要么 changeFailed，无半成品。
  - 批量全或无：`connect_many` 一条 `<group>` 原子提交（最接近"事务"的点，但只在单个工具的一批操作内）。
  - 单步回退：`applyChange` 设 `setUndoable(true)` + `undo()/redo()`（Ptolemy `UndoStackAttribute`）。
  - 失败补偿：`GroupIntoCompositeTool` wrap 失败时主动 delete 半成品（saga 的 compensation 思想）。
  - 预演（不提交）：`dryRunTool`（见下方专条），是事务"预览"半，且当前未接线。
  - 划边界话术："这是工具级/单进程原子性 + 单步回退 + 补偿，不是带隔离性/持久性的 ACID/分布式事务；完整事务可在这套基元上再往上搭。"
- **dryRunTool = 事务的"预览半"，当前是 latent primitive（休眠基元）** ⚠️：
  - 机制：`ToolRegistry.dispatch` 见参数带 `dryRun`/`_dryRun`（且非 validate）→ 转 `session.dryRunTool()` → **克隆当前模型到临时 session**（`exportMoml()`→新 session `loadMoml`）→ 在**副本**上执行工具 + 跑 validate → 返回 `{committed:false, toolResult, validation, modelContext}` → `finally` 销毁副本。真身零 mutate。
  - 设计用途：危险操作（delete/group）前预演、agent 的 what-if 探针、"预演→确认→提交"事务式工作流的底层基元（注释原文 "primitive for dry-run / transaction workflows"）。
  - ⚠️ **诚实边界**：全仓库**没有任何调用方设这个 flag**（RestRoutes/tools/test/frontend 都没有）——通道与实现都在，但**未接到 pipeline/前端**。它**只有预览半、没有 commit 半**（不存在把副本提交回真身的路径）。所以它**不是事务**，是"预览基元"。
  - 能说：设计思路、克隆-执行-校验-销毁机制、和 undo 的互补（做之前预览 vs 做之后回退）。**别说**"pipeline 靠它做安全执行/降低错误率/前端确认框基于它"——会被"给我看调用处"问穿。
- **错误设计（Error as data）**：好的工具错误应该是"可行动的"——告诉 LLM 哪里错、正确选项是什么、下一步怎么做，而不是死胡同 message。这是降低无效重试的关键。
- **function calling 参数校验**：JSON Schema 约束 + 运行时校验双保险。schema 挡住类型错误，运行时校验挡住语义错误（端口不存在、方向反）。

### 高频追问

- **Q：didYouMean 的算法复杂度和排序规则？**
  A：Levenshtein DP + 4 桶排序（精确/前后缀/子串/编辑距离），top-3，距离阈值 3。方向感知（要 input 不推 output）。
- **Q：批量回滚怎么保证原子？MoML apply 中途失败呢？**
  A：先全 preflight，任一错整批拒（返回 rejected[] 带 didYouMean + accepted[].status=would-apply）；全过才拼一条 `<group>` 提交；若 Ptolemy 拒这条 group，则整批不落地。严格说这是 **abort-before-commit（还没写就拦下）+ 原子 group 提交**，不是"写一半再撤"。
- **Q（易踩坑）：你这算事务吗？dry-run 是事务功能吗？**
  A（主动划边界）："不是完整事务。我没有跨多操作的 begin/commit/rollback 边界或事务管理器。我有的是工具级原子性（单次 MoMLChangeRequest + connect_many 的 `<group>` 批量全或无）、单步 undo/redo 回退、以及 group 失败时的补偿清理。dry-run 是我为'预演→确认→提交'这类事务式工作流设计的**预览基元**——它克隆模型试跑不提交，但只有预览半、没有 commit 半，而且当前还没接到任何调用方。完整事务是可以在这些基元上再往上搭的。"
- **Q：幂等和"防振荡"的区别？**
  A：幂等是单次工具的性质（重复调不报错）；防振荡是跨多次调用的模式检测（delete→add 循环），在 `AgentLoop` 层。两者叠加。
- **Q：错误上下文为什么要"嵌进 message"？**
  A：双保险。结构化字段给会解析的 LLM，message 文本给不解析的，两条路都能让模型看到 "Did you mean Sum.minus?"。
- **Q：classFailures 归因怎么避免错杀？**
  A：按失败 payload 的 `issues[*].field` 只归咎出错的那一端（connect 的 from/to），一次 dispatch 对同一类只记一次，只记真正归咎的类（记 Expression 不记 Ramp）。

## 条目 9 — 确定性调度

**简历原文**：SDF Director 通过拓扑分析与消费/产出率匹配，在编译期计算出各 Actor 的执行次数与顺序；DE Director 按全局时间戳排序事件，相同拓扑与输入得到一致执行轨迹。

## 条目 10 — 分阶段执行协议

**简历原文**：Actor 标准化为 prefire（前置条件检查）→ fire（计算执行）→ postfire（状态提交）三段式生命周期，Director 在各阶段之间注入调度逻辑，横切关注点可在不修改 Actor 的前提下统一施加。

### 必备知识点
- **prefire**：检查是否满足触发条件（输入是否就绪），返回能否 fire。
- **fire**：纯计算，可能被多次调用，不应提交不可逆状态。
- **postfire**：提交状态、推进到下一步，返回是否继续。
- **核心思想**：把"检查/计算/提交"分离，Director 在阶段间统一施加调度，AOP（横切关注点）思想。

### 迁移到 Agent（必讲，强类比）
三段式 → 我的工具调用生命周期：**preflight 校验（prefire）→ dispatch 执行（fire）→ trace 记录/结果提交（postfire）**。校验、幂等、防振荡这些横切关注点在 `ToolRegistry.dispatch` 和 `AgentLoop` 里统一施加，**不改每个具体工具**。这就是 prefire/postfire 让"横切关注点统一施加"的思想落地。

### 高频追问
- Q：为什么 fire 要幂等/无副作用，postfire 才提交？A：Director 可能试探性多次 fire（如求不动点），状态提交必须放在确定推进的 postfire。
- Q：对应你的工具设计？A：preflight 只读校验（可反复），真正 mutation 在 dispatch，结果落 trace——关注点分离让我能统一加校验/防振荡而不碰工具本身。

### 电梯陈述（背）
同一角色的论元语义分布是**多模态**的（一个角色可对应多种表达），单原型表示不够。我在**超球面**上用**动态多原型**表示每个角色，并**联合建模论元间依赖与角色一致性约束**，提升抽取的准确与一致性。

### 必备知识点
- **原型（prototype）学习**：每类用一个/多个代表向量，按距离分类。多原型解决"类内多模态分布"。
- **为什么超球面**：把表示归一化到单位球面，用角度/余弦度量，训练更稳、类间更可分（对比学习常用）。
- **动态多原型**：原型数量/位置随数据自适应，而非固定。
- **论元依赖 + 角色一致性**：论元之间不是独立的（一个事件里的角色相互约束），联合建模避免自相矛盾的抽取。

### 高频追问 + 迁移
- Q：为什么用超球面而不是欧氏空间？A：归一化后角度度量更稳、利于对比学习、缓解 magnitude 干扰。

产物在 `eval/results/`：逐次 CSV + 对照汇总 Markdown 表。

**它怎么工作（能被追问到实现细节，所以要懂）**：
- 每条任务：`POST /sessions` 建会话 → `POST /sessions/{id}/agent/chat`（请求体带 `mode` 强制 pipeline/single，**无需重启即可 A/B**）→ 解析返回的 `trace.steps` → 再单独 `POST /run` 做**客观**"模型真能跑吗"检查 → `DELETE` 会话。
- 指标全部来自真实字段：`trace.success`、`steps[].kind`(tool_call/tool_result)、`tool_result` 的 `arguments.ok`(失败调用)、`arguments.antiThrash`(反振荡命中)、`arguments.data.noop`(幂等命中)、文本里 `[Phase 1/4: skipped …]`(builder 跳过，启发式)。

- **不能说**：❌ 精确到小数的"准确率 +37%"；❌ "逐层防线消融实验"（没有 feature flag，做不了）；❌ 任何 token 数字（接口没有）。
